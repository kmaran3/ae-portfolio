# E-Commerce Data Mart — Analytics Engineer Portfolio Project

> **Stack:** dbt · BigQuery · Looker Studio · Python (jafgen) · GitHub  
> **Goal:** A production-style, fully modeled data mart built on synthetic e-commerce data — with a 3-layer dbt architecture, automated data quality tests, full lineage documentation, and a business dashboard on top.  
> **Why this project:** It mirrors exactly what Analytics Engineers build day to day. The dbt docs lineage graph, test coverage, and mart design are what interviewers look for — not the data itself.

### 3.3 Update `dbt_project.yml`

Open `jaffle_shop/dbt_project.yml` and replace the `models:` section with:

```yaml
models:
  jaffle_shop:
    staging:
      +materialized: view
    intermediate:
      +materialized: view
    marts:
      +materialized: table
```

This tells dbt:
- Staging and intermediate models are **views** (no storage cost, always fresh)
- Mart models are **tables** (materialized for fast BI queries)

### 3.4 Verify Connection

```bash
dbt debug
```

Every line should say `OK`. If BigQuery connection fails, double-check your `gcp_key.json` path in `profiles.yml` (located at `~/.dbt/profiles.yml`).

### 3.5 Install `dbt_utils` Package

`dbt_utils` is the most widely used dbt package in enterprise environments. It provides macros for surrogate key generation, date spine creation, and additional test types. Using it signals real-world familiarity.

Create `jaffle_shop/packages.yml`:

```yaml
packages:
  - package: dbt-labs/dbt_utils
    version: [">=1.0.0", "<2.0.0"]
```

Then install it:

```bash
dbt deps
```

You'll use `dbt_utils.generate_surrogate_key()` in the `fct_revenue_daily` mart later. This is the production pattern for creating stable primary keys on aggregated models that have no natural single-column key.

> **Enterprise pattern:** In real deployments, `packages.yml` is version-pinned and committed to the repo. It gets installed as part of CI/CD before `dbt run` executes. Treat it as a required dependency, not an add-on.

## Phase 4: Build Staging Models (Day 2 continued)

Staging models are 1:1 with raw tables. The only work here is renaming columns to a consistent convention, casting data types, and adding `_at` suffixes to timestamps. No business logic yet.

### 4.0 `models/staging/sources.yml` — Declare Your Raw Tables

Before writing a single staging model, you need to declare your raw BigQuery tables as **dbt sources**. This is the correct pattern — never hardcode a full BigQuery path like `` `project.dataset.table` `` directly in a staging model.

Why this matters:
- It adds your raw tables to the **lineage graph** so interviewers can see the full data flow from raw → staging → marts
- It enables `dbt source freshness` — you can alert when raw data stops arriving
- It's how every enterprise dbt project is structured; hardcoded paths are a code smell

```yaml
version: 2

sources:
  - name: raw
    database: krishna-ae-portfolio
    schema: raw
    description: "Raw tables loaded directly from jafgen CSV exports into BigQuery."

    freshness:
      warn_after: {count: 24, period: hour}
      error_after: {count: 48, period: hour}
    loaded_at_field: ordered_at   # used by dbt to check freshness on orders

    tables:
      - name: raw_customers
        description: "Raw customer records from jafgen."
      - name: raw_orders
        description: "Raw order transactions. The loaded_at_field freshness check runs on this table."
      - name: raw_order_items
        description: "Raw line items — one row per product per order."
      - name: raw_products
        description: "Raw product catalog."
      - name: raw_stores
        description: "Raw store locations."
```

> **Note on freshness:** `loaded_at_field` is set to `ordered_at` on the source level here for simplicity. In production you'd typically have a separate `_etl_loaded_at` timestamp column added by your ingestion tool (Fivetran, Airbyte, etc.) that reflects when the *pipeline* ran, not when the order was placed. Keep this distinction in mind — it's a common interview question.

After creating this file, run:
```bash
dbt source freshness
```
It will likely warn or error since jafgen data is static (no new rows arriving), but the command should execute without crashing, confirming your source configuration is valid.

## Phase 5: Build Intermediate Models (Day 3)

Intermediate models join staging models together and apply business logic. They are never exposed directly to BI tools — they exist to keep mart models clean and readable.

### 5.1 `models/intermediate/int_orders_with_customers.sql`

```sql
with orders as (
    select * from {{ ref('stg_orders') }}
),

customers as (
    select * from {{ ref('stg_customers') }}
),

joined as (
    select
        orders.order_id,
        orders.store_id,
        orders.status,
        orders.subtotal_dollars,
        orders.tax_paid_dollars,
        orders.subtotal_dollars + orders.tax_paid_dollars  as total_dollars,
        orders.ordered_at,
        customers.customer_id,
        customers.customer_name,
        customers.created_at                               as customer_created_at
    from orders
    left join customers using (customer_id)
)

select * from joined
```

## Phase 6: Build Mart Models (Day 3 continued)

Mart models are the final, business-facing tables. These are what get connected to Looker Studio. Each one should answer a specific business question.

### 6.1 `models/marts/core/fct_orders.sql`

The order-level fact table. One row per order with all key metrics.

```sql
with orders as (
    select * from {{ ref('int_orders_with_customers') }}
),

final as (
    select
        order_id,
        customer_id,
        customer_name,
        store_id,
        status,
        subtotal_dollars,
        tax_paid_dollars,
        total_dollars,
        ordered_at,
        date(ordered_at)                                    as order_date,
        format_date('%Y-%m', date(ordered_at))              as order_month,

        -- Useful flags
        case when status = 'completed' then 1 else 0 end    as is_completed,
        case when status in ('return_pending', 'returned')
             then 1 else 0 end                              as is_returned
    from orders
)

select * from final
```

### 6.2 `models/marts/core/dim_customers.sql`

Customer dimension with lifetime value and order history — the most important table for retention and LTV analysis.

```sql
with orders as (
    select * from {{ ref('fct_orders') }}
),

customers as (
    select * from {{ ref('stg_customers') }}
),

customer_orders as (
    select
        customer_id,
        count(order_id)                                     as total_orders,
        sum(total_dollars)                                  as lifetime_value_dollars,
        min(ordered_at)                                     as first_order_at,
        max(ordered_at)                                     as most_recent_order_at,
        sum(is_completed)                                   as completed_orders,
        sum(is_returned)                                    as returned_orders
    from orders
    group by customer_id
),

final as (
    select
        customers.customer_id,
        customers.customer_name,
        customers.created_at                                as customer_created_at,
        coalesce(customer_orders.total_orders, 0)           as total_orders,
        coalesce(customer_orders.lifetime_value_dollars, 0) as lifetime_value_dollars,
        customer_orders.first_order_at,
        customer_orders.most_recent_order_at,
        coalesce(customer_orders.completed_orders, 0)       as completed_orders,
        coalesce(customer_orders.returned_orders, 0)        as returned_orders,

        -- Segmentation
        case
            when customer_orders.total_orders >= 5 then 'loyal'
            when customer_orders.total_orders >= 2 then 'repeat'
            when customer_orders.total_orders = 1 then 'one_time'
            else 'no_orders'
        end                                                 as customer_segment
    from customers
    left join customer_orders using (customer_id)
)

select * from final
```

### 6.3b `models/marts/core/fct_order_items_enriched.sql`

This model closes the loop on the intermediate layer — `int_order_items_with_products` was built but never surfaced to a mart in the original plan. This table answers product-level questions: which products drive the most revenue? Which categories have the highest return rates?

```sql
with order_items as (
    select * from {{ ref('int_order_items_with_products') }}
),

orders as (
    select
        order_id,
        customer_id,
        store_id,
        status,
        is_completed,
        is_returned,
        order_date,
        order_month
    from {{ ref('fct_orders') }}
),

final as (
    select
        order_items.order_item_id,
        order_items.order_id,
        order_items.product_id,
        order_items.product_name,
        order_items.product_type,
        order_items.price_dollars,
        orders.customer_id,
        orders.store_id,
        orders.status,
        orders.is_completed,
        orders.is_returned,
        orders.order_date,
        orders.order_month
    from order_items
    left join orders using (order_id)
)

select * from final
```

> **Enterprise pattern:** Notice that `fct_order_items_enriched` joins to `fct_orders` (a mart model) using `ref()`, not back to the intermediate layer. This is intentional — marts can reference other marts when the source mart is a stable, tested output. Avoid skipping layers in the other direction (e.g., don't reference a staging model from a mart).

### 6.4 `models/marts/finance/fct_revenue_daily.sql`

Daily revenue aggregated by store — the primary table for the dashboard trend chart. Note the use of `dbt_utils.generate_surrogate_key()` to create a stable primary key on this aggregated model, since no single column uniquely identifies a row.

```sql
with orders as (
    select * from {{ ref('fct_orders') }}
    where is_completed = 1
),

daily as (
    select
        order_date,
        order_month,
        store_id,
        count(order_id)             as orders_placed,
        sum(total_dollars)          as gross_revenue_dollars,
        sum(subtotal_dollars)       as net_revenue_dollars,
        sum(tax_paid_dollars)       as tax_collected_dollars,
        avg(total_dollars)          as avg_order_value_dollars
    from orders
    group by order_date, order_month, store_id
),

final as (
    select
        {{ dbt_utils.generate_surrogate_key(['order_date', 'store_id']) }} as revenue_daily_id,
        order_date,
        order_month,
        store_id,
        orders_placed,
        gross_revenue_dollars,
        net_revenue_dollars,
        tax_collected_dollars,
        avg_order_value_dollars
    from daily
)

select * from final
```

### 6.6 `models/marts/marketing/fct_customer_cohorts.sql`

Cohort retention table — the most technically impressive model. Shows what % of customers from each signup month are still ordering N months later.

```sql
with customers as (
    select
        customer_id,
        format_date('%Y-%m', date(customer_created_at))  as cohort_month
    from {{ ref('dim_customers') }}
),

orders as (
    select
        customer_id,
        format_date('%Y-%m', order_date)                 as order_month,
        order_date
    from {{ ref('fct_orders') }}
    where is_completed = 1
),

cohort_orders as (
    select
        customers.customer_id,
        customers.cohort_month,
        orders.order_month,
        date_diff(
            parse_date('%Y-%m', orders.order_month),
            parse_date('%Y-%m', customers.cohort_month),
            month
        )                                                as months_since_signup
    from customers
    left join orders using (customer_id)
),

cohort_sizes as (
    select
        cohort_month,
        count(distinct customer_id) as cohort_size
    from customers
    group by cohort_month
),

retention as (
    select
        cohort_orders.cohort_month,
        cohort_orders.months_since_signup,
        count(distinct cohort_orders.customer_id)        as active_customers
    from cohort_orders
    where months_since_signup is not null
      and months_since_signup >= 0
    group by cohort_month, months_since_signup
),

final as (
    select
        retention.cohort_month,
        retention.months_since_signup,
        retention.active_customers,
        cohort_sizes.cohort_size,
        round(
            100.0 * retention.active_customers / cohort_sizes.cohort_size,
            1
        )                                                as retention_rate_pct
    from retention
    left join cohort_sizes using (cohort_month)
)

select * from final
order by cohort_month, months_since_signup
```

## Architecture Decisions

**Why `{{ source() }}` instead of hardcoded table paths?**  
Using the `source()` macro connects raw tables to the dbt DAG, enabling lineage tracking from raw → staging → marts. It also unlocks `dbt source freshness` checks so pipelines alert when data stops arriving. Hardcoding paths breaks lineage and is considered an anti-pattern in production.

**Why views for staging/intermediate and tables for marts?**  
Views have no storage cost and are always fresh — ideal for the cleaning and joining layers where we want changes to propagate immediately. Mart tables are materialized because BI tools query them repeatedly; a table is far faster than re-executing a multi-join view at query time.

**Why a surrogate key on `fct_revenue_daily`?**  
Aggregated models have no natural single-column primary key. Using `dbt_utils.generate_surrogate_key(['order_date', 'store_id'])` creates a stable, testable key from the composite of columns that uniquely identify a row. This is the production pattern for any aggregated fact table.

**Why does `fct_order_items_enriched` join to `fct_orders` rather than the intermediate layer?**  
Marts can reference other marts via `ref()` when the upstream mart is stable and tested. Joining to `fct_orders` means the product-level fact inherits the same business logic (is_completed flag, order_month calculation) without duplicating it.

- Surrogate key uniqueness enforced on aggregated mart
- Source freshness configured on raw tables
- All tests passing on 2 years of synthetic data (~100k+ rows)
